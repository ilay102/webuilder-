## 1. WHAT IS THIS PROJECT?

**Webuilder** is a fully automated B2B website-builder SaaS for Israeli small businesses (dental clinics, accountants, lawyers). The workflow:

1. **JJ** (WhatsApp sales agent) finds leads → qualifies → closes
2. **Carti** (builder agent) auto-builds a personalized demo site from a queue
3. Client sees demo → fills intake form → `content.json` updates → git push → Vercel rebuilds in ~60s
4. Client sends photos via WhatsApp → auto-uploaded + slot-detected → git push → site updates
5. Client pays (Stripe — wiring next) → site locks, goes fully live
6. Every client gets a **unique design** from the variance engine — same template, completely different look + AI copy

**Flagship gold standard demo:** `https://webuilder-liart.vercel.app/cohen-dental-v2`
This is the reference. Every new demo must match or exceed this quality.

## 5. AUTOMATED VARIANCE ENGINE

> **Goal:** Same Next.js template → visually and textually distinct per client. Fully automated at demo creation. Zero manual work after `new-demo.ts` runs.

### 5a. Design Packs (`lib/design-packs.ts`)

12 coordinated "brand personalities". Each defines:

| Field | Controls |
|---|---|
| `colors.*` | Full 8-token palette (primary, accent, bg, bgAlt, white, text, muted, light) |
| `fonts.heading/body` | Google Fonts family + weights |
| `radius` | Base border-radius (2px → 24px) |
| `heroOverlay.blend` | CSS `mix-blend-mode` for photo tinting |
| `heroOverlay.gradient` | Color gradient layered over photo |
| `heroOverlay.opacity` | Overlay strength |
| `photoFilter` | CSS `filter` applied to ALL photos site-wide |
| `vibe` | Short phrase fed to Gemini so copy tone matches visual tone |

**The 12 packs:**
1. `forest-serif-soft` — sage green, warm, nurturing
2. `navy-sans-sharp` — deep blue, confident, sharp
3. `coral-warm-friendly` — coral/orange, family-friendly
4. `charcoal-mono-tech` — near-greyscale, clinical precision
5. `teal-modern-airy` — teal, wellness-forward, fresh
6. `ivory-editorial-classic` — ivory/burgundy, boutique, Cormorant font
7. `midnight-luxe-gold` — **dark theme**, gold accents, luxury
8. `sand-minimal-zen` — sand/terracotta, spa-like
9. `rose-gentle-feminine` — rose/mauve, cosmetic/aesthetic
10. `emerald-pro-trust` — emerald, established professional
11. `amber-friendly-family` — amber/gold, neighbourhood clinic
12. `slate-tech-clean` — slate/cyan, high-tech, IBM Plex Sans

### 5b. Vibe Blend CSS Logic

The hero section layers **three stacked elements**:
```
[1] <img>  — pool photo, CSS filter applied (PACK.photoFilter)
[2] <div>  — PACK color overlay, mix-blend-mode: [multiply|soft-light|overlay|color]
             "The Vibe Blend" — tints any photo to match brand palette
[3] <div>  — Readability wash (bg color + opacity hex suffix)
             gradient: linear-gradient(to right, ${bg}e8 38%, ${bg}30 70%, transparent 100%)
```
Result: any stock dental photo looks like it was shot for that specific brand's palette.

### 5c. Deterministic Hash (`lib/variance.ts`)
```
slug + city → FNV-1a hash → modulo 12 → packId
Same slug always = same design (reproducible)
Different slugs = different designs (guaranteed)
```

### 5d. AI Copy (`lib/generate-copy.ts`)

Runs **once at build time** inside `new-demo.ts`. Result baked into `content.json → copy{}`.
- Uses Gemini 2.5 Flash, temperature 0.85 (high variety)
- Loaded from `.env` via `dotenv` at script startup
- **Fallback pool**: 5 hand-crafted Hebrew headline sets, picked by `FNV hash(businessName + city) % 5`
  → Different demos get different fallback headlines even without API
- Strips markdown code fences from Gemini response (handles ```json wrapping)
- Falls back gracefully on 503/non-200/parse errors

### 5e. Full Pipeline (inside `new-demo.ts`)
```
new-demo.ts --route, --name, --city, --template, etc.
│
├─ 1. Load .env (dotenv) → GEMINI_API_KEY available
├─ 2. pickVariant(slug, template, city) → packId
├─ 3. getPack(packId) → full pack object
├─ 4. allocateImage('hero', slug) → pool image (FIFO)
├─ 5. allocateImage('patient', slug) → pool image (FIFO)
├─ 6. generateCopy({ name, city, packVibe, services }) → copy object
├─ 7. Build content.json: biz + Hebrew services/stats/testimonials + pool photos + design + copy
├─ 8. Copy dental/page.tsx to app/[route]/page.tsx
├─ 9. Register domain (if --domain) → domains.json + Vercel API
├─ 10. git add + commit + pull --rebase + push  (pool-state.json included)
│      On git failure → freeImages(slug) rollback
└─ 11. POST /api/clients on VPS → clients.json entry
```

## 6. NEW: IMAGE POOL LIBRARY MODEL

> **Why:** Automated demos need real dental photos. We can't use random stock images or the same photo for every client. The Library Model gives each demo a unique, high-quality photo from a curated pool, with full lifecycle management.

### 6a. Pool Structure

```
public/pool/dental/
├── heroes/    ← 5 clinic environment photos
└── patients/  ← 9 patient/smile portrait photos
```

**pool-state.json** (repo root, git-tracked):
```json
{
  "updatedAt": "2026-04-18T...",
  "images": [
    {
      "id": "heroes/Gemini_Generated_Image_xxx.png",
      "path": "/pool/dental/heroes/Gemini_Generated_Image_xxx.png",
      "type": "hero",
      "status": "in-use",
      "assignedTo": "test-clinic-1",
      "assignedAt": "2026-04-17T...",
      "addedAt": "2026-04-17T..."
    }
  ]
}
```

**Statuses:**
- `available` — free to allocate to next demo
- `in-use` — assigned to a client (demo created, not yet paid)
- `locked` — client paid; image is permanently theirs

### 6b. `lib/pool-manager.ts` — The Lifecycle Engine

Key functions:
```ts
allocateImage(type: 'hero'|'patient', slug: string): PoolImage
  // FIFO: picks oldest available image of that type
  // Sets status → 'in-use', assignedTo = slug
  // Throws detailed error if pool exhausted

freeImages(slug: string): string[]
  // Releases ALL images assigned to slug back to 'available'
  // Called on: git push failure (rollback), client churn (delete)

freeImageByPath(slug: string, imagePath: string): boolean
  // Releases ONE specific image
  // Called when client uploads their own photo (replaces the pool image)

lockImages(slug: string): string[]
  // Sets status → 'locked' for all images assigned to slug
  // Called on: Stripe payment confirmed

poolStats(): { hero: {...}, patient: {...} }
  // Returns counts per type: available / in-use / locked / total
```

### 6c. Full Image Lifecycle

```
1. DEMO CREATED (new-demo.ts)
   allocateImage('hero', slug)    → status: available → in-use
   allocateImage('patient', slug) → status: available → in-use
   [On git push failure] freeImages(slug) → rollback to available

2. CLIENT UPLOADS OWN PHOTO (intake-server.js /api/photo)
   Detect if old slot was a pool image (path.startsWith('/pool/'))
   freePoolImage(slug, oldPath) → status: in-use → available
   Save new client photo to /public/clients/[slug]/[slot].jpg
   Update content.json with new path

3. CLIENT CHURNS (server/index.js DELETE /api/clients/:slug)
   _freeImages(slug) → all in-use images → available
   (locked images stay locked — already paid for)

4. CLIENT PAYS (Stripe webhook → POST /api/clients/:slug/lock)
   _lockImages(slug) → all in-use images → locked
   Update client status: trial → active
   [TODO: trigger JJ to send congratulations + domain setup instructions]
```

### 6d. Pool Initialization

Run once to register all images:
```bash
npm run pool:init
```
Script: `scripts/pool-init.ts` — scans `public/pool/dental/heroes/` and `public/pool/dental/patients/`, registers each file in pool-state.json (idempotent).

Check pool health anytime:
```bash
npm run pool:stats
# Output: { hero: { available: 1, inUse: 4, locked: 0, total: 5 }, patient: { ... } }
```

## 7. THE DENTAL PAGE TEMPLATE (`app/dental/page.tsx`)

**This is the single source of truth for all client site layouts.** Every demo's `page.tsx` is a COPY of this file made at demo-creation time.

### Key Architecture Decisions:

**RTL + Hebrew:**
- Root div has `dir="rtl"` for correct Hebrew layout
- Hero content block has `direction: 'ltr', textAlign: 'left'` so text reads naturally on the left side
- **CRITICAL RTL FLEX RULE:** In `dir="rtl"`, `flex-end` = physical LEFT, `flex-start` = physical RIGHT. Always verify visually after RTL changes.

**Hero gradient direction:**
- `linear-gradient(to right, ...)` → fade covers LEFT side where text lives ✅

**Hero layout:**
- `hero-inner`: `justifyContent: 'flex-end'` (= physical left in RTL), NO maxWidth/margin constraints (full inset-0)
- `hero-content`: `maxWidth: 560`, `direction: ltr`, `textAlign: left`

**Photo cast:**
```tsx
const photos = content.photos as Record<string, any>;
// Needed because template content.json has null photos (TypeScript strict mode)
```

**All hardcoded text is Hebrew:**
- Nav: שירותים | אודות | המלצות | קבע תור
- Sections: מה אנחנו מציעים, אודות המרפאה, תוצאות, המלצות
- CTA: מוכנים להתחיל?, החיוך המושלם שלך במרחק תור אחד
- Footer: שירותים | המרפאה | צור קשר
- Chatbot greeting in Hebrew

## 13. KNOWN ISSUES / WATCH OUT FOR

1. **RTL flex direction is REVERSED:**
   - `flex-end` = physical LEFT ✅
   - `flex-start` = physical RIGHT ❌
   - Always test visually after changing hero layout in RTL context.

2. **Template photos must stay null:**
   - `app/dental/content.json` photos are all `null`
   - Client WhatsApp test uploads go to `dental/` slug — DON'T use dental as a real client
   - If template gets contaminated, new demos inherit the photo. Fix: reset photos to null, commit.

3. **new-demo.ts copies page.tsx at creation time:**
   - If you update `dental/page.tsx`, existing demos have the old version
   - To update all demos: `cp app/dental/page.tsx app/[slug]/page.tsx` for each + commit
   - New demos created after the update will automatically get the new version

4. **VPS git push race condition:**
   - When VPS runs `new-demo.ts` AND local pushes at same time → push rejected
   - Fix: always `git stash && git pull --rebase && git push && git stash pop` locally

5. **Gemini copy 503 errors:**
   - Free tier rate limits → falls back to pool gracefully
   - Not a code bug — just API quota

6. **Pool exhaustion:**
   - When `allocateImage` throws, the demo fails and images are rolled back
   - Monitor with `npm run pool:stats` before batch demo creation
